12.dim tables 


---

# 1. Problem we discovered (schema issue)

Initially our DBT models were creating objects in the **same schema as raw tables**.

Example:

```text
Database: movielens
Schema: raw
```

Inside this schema we had:

```text
raw_movies
raw_ratings
raw_tags
```

But DBT also started creating:

```text
src_movies
src_ratings
dim_movies
```

So everything was mixed in the **same schema**.

This is **bad practice**.

---

# 2. Why mixing schemas is bad

If raw data and transformed data are together, it becomes messy.

Example bad structure:

```text
movielens.raw

raw_movies
raw_ratings
src_movies
src_ratings
dim_movies
dim_users
```

Problems:

* Hard to understand pipeline
* Risk of accidentally modifying raw data
* Production pipelines become messy
* Hard for teams to manage

---

# 3. Correct warehouse structure

In real companies schemas are separated.

Example:

```text
movielens.raw        → raw ingestion data
movielens.dev        → dbt models
movielens.analytics  → BI tables
```

Architecture becomes:

```text
Source
   ↓
RAW schema
   ↓
DBT staging models
   ↓
DBT dimension / fact tables
   ↓
Analytics dashboards
```

---

# 4. What we changed

We changed the DBT **profile configuration**.

Instead of writing models into:

```text
schema: raw
```

we changed it to:

```text
schema: dev
```

So DBT now writes everything into a **development schema**.

---

# 5. New warehouse structure

Now the warehouse looks like this:

```text
Database: movielens

Schema: raw
--------------------------------
raw_movies
raw_ratings
raw_tags

Schema: dev
--------------------------------
src_movies
src_ratings
src_tags
dim_movies
dim_users
dim_genome_tags
```

Much cleaner.

---

# 6. What we already had before

From earlier steps we already created **staging models**.

These were views.

Example:

```text
src_movies
src_ratings
src_tags
src_links
src_genome_tags
src_genome_scores
```

All of these are **views** created by DBT.

They simply clean raw data.

Architecture so far:

```text
Raw tables
   ↓
Staging views
```

---

# 7. Now we started building dimension tables

Next layer of warehouse is **dimension tables**.

So we created a new folder:

```text
models/dim
```

Inside this folder DBT configuration says:

```text
materialized = table
```

So everything inside `dim` becomes **tables**.

---

# 8. What we built

We created **three dimension tables**

```text
dim_movies
dim_users
dim_genome_tags
```

These tables are built **using staging models**.

This is where DBT’s **ref() dependency system** is used.

---

# 9. What dim_movies does

Conceptually:

```text
src_movies (staging view)
        ↓
dim_movies (clean dimension table)
```

The model reads data from:

```text
src_movies
```

Then performs transformations like:

* formatting title
* cleaning text
* splitting genres
* renaming columns

Result:

```text
dim_movies
-------------
movie_id
movie_title
genre_array
```

This becomes the **movie dimension table**.

---

# 10. What dim_users does (important concept)

This one is more interesting.

User data actually appears in **two datasets**:

```text
ratings
tags
```

Both tables contain `user_id`.

So we want a **complete list of all users**.

To do this the model:

1. extracts user IDs from ratings
2. extracts user IDs from tags
3. merges them together
4. removes duplicates

Conceptually:

```text
src_ratings
      ↓
get user_ids

src_tags
      ↓
get user_ids

combine both
      ↓
remove duplicates
      ↓
dim_users
```

Final table:

```text
dim_users
-----------
user_id
```

This ensures **every user in the system appears once**.

---

# 11. What dim_genome_tags does

This model creates a dimension for **tag descriptions**.

Data source:

```text
src_genome_tags
```

Transformation:

* clean tag names
* format text
* rename columns

Conceptually:

```text
src_genome_tags
       ↓
clean text
       ↓
dim_genome_tags
```

Final structure:

```text
dim_genome_tags
----------------
tag_id
tag_name
```

---

# 12. Important concept shown here

This is where DBT becomes powerful.

Each dimension model **references staging models**.

Example flow:

```text
src_movies
      ↓
dim_movies

src_ratings + src_tags
      ↓
dim_users

src_genome_tags
      ↓
dim_genome_tags
```

This creates a **dependency graph**.

DBT automatically understands execution order.

---

# 13. What DBT run does now

When you run:

```text
dbt run
```

DBT executes models in correct order.

Example execution:

```text
src_movies
src_ratings
src_tags
src_genome_tags
      ↓
dim_movies
dim_users
dim_genome_tags
```

So staging runs first, then dimensions.

---

# 14. Final warehouse architecture now

After this step the warehouse looks like this:

```text
RAW LAYER
--------------------------------
raw_movies
raw_ratings
raw_tags
raw_links
raw_genome_tags
raw_genome_scores

STAGING LAYER (views)
--------------------------------
src_movies
src_ratings
src_tags
src_links
src_genome_tags
src_genome_scores

DIMENSION LAYER (tables)
--------------------------------
dim_movies
dim_users
dim_genome_tags
```

Next step later will be **fact tables**.

---

# 15. Why this design is used everywhere

This layered design is used in companies like:

* Netflix
* Uber
* Airbnb
* Spotify

Typical pipeline:

```text
Data Sources
     ↓
Raw tables
     ↓
Staging models
     ↓
Dimension tables
     ↓
Fact tables
     ↓
BI dashboards
```

DBT mainly handles the **transformation layer**.

---

# 16. One very important thing you probably noticed

We are **not referencing raw tables directly anymore**.

Instead we always use:

```text
staging models → dimension models
```

This keeps the pipeline **clean and modular**.
