17.Materialized View

A materialized view is a saved query result stored as a table.

👉 Unlike a normal view, it stores the data physically, not just the SQL.

Normal View vs Materialized View
Type	What happens
View	Runs query every time
Materialized View	Stores result once
Example
Normal View
CREATE VIEW sales_view AS
SELECT product_id, SUM(amount)
FROM sales
GROUP BY product_id;

Every time you query it → recalculates.

Materialized View
CREATE MATERIALIZED VIEW sales_mv AS
SELECT product_id, SUM(amount)
FROM sales
GROUP BY product_id;

👉 Result is stored like a table.

What it does

Pre-computes data

Stores results

Makes queries faster

Example use case

Dashboard showing:

total sales

monthly revenue

Instead of recalculating every time →
materialized view already has the result.

Important point

Materialized view needs refresh to get new data.

Example:

data updated → view not auto-updated (depends on system)

you refresh → data becomes latest

Simple analogy

View → like a live calculation
Materialized view → like a saved result

One-line answer

Materialized view is a stored, precomputed query result used to make queries faster.

why not use a table instead , why keep refreshing alwasys

Good question — this is exactly where many people get confused.

Short answer

A materialized view = auto-maintained table based on a query.
A normal table = you manage everything manually.

Why not just use a table?

You can use a table, but then you must:

write ETL/DBT logic

schedule updates

handle dependencies

ensure data is always correct

👉 More work, more chances of mistakes.

What materialized view does for you

It automatically keeps data updated based on the query.

Example:

SELECT product_id, SUM(amount)
FROM sales
GROUP BY product_id;

Instead of:

writing pipeline

updating table daily

Materialized view:

stores result

refreshes when needed

Key difference
Feature	Table	Materialized View
Data storage	Yes	Yes
Auto update	❌ No	✅ Yes (auto/refresh)
Setup effort	High	Low
Control	Full	Limited
When to use table

Use table when:

complex transformations (DBT pipelines)

business logic layers (fact/dim tables)

need full control

Example:
fact_sales, dim_customers

When to use materialized view

Use it when:

simple aggregation

frequently used query

need fast performance

Example:

dashboard metrics

daily totals

Why refresh is needed

Because data changes:

Sales table:

Day 1 → 1000
Day 2 → 1500

Materialized view:

still shows old value until refreshed

Refresh = recalculate and update stored result

Simple analogy

Table → you cook food manually every time
Materialized view → food is pre-cooked but needs reheating (refresh)

Final simple answer

We use materialized views instead of tables when we want fast, precomputed data without managing full pipelines — refresh just keeps it updated.
