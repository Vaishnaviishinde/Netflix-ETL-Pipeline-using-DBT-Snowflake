18.Seeds
---

# 1. First: What are “Seeds” (Concept)

### Simple meaning

**Seeds = CSV file → DBT converts it into a table in your warehouse**

---

### Why does this exist?

Normally to create a table you need:

* upload file to S3
* create table
* load data

Too much work.

So DBT says:

> “Just drop a CSV → I’ll create the table for you”

---

# 2. In your project (what actually happened)

You created:

```text
seeds/seed_movie_release_date.csv
```

Then ran:

```bash
dbt seed
```

DBT did:

```text
CSV file → Snowflake table
```

Now this exists like a normal table:

```text
seed_movie_release_date
```

---

# 3. How you used it

In your model:

```sql
WITH seed_dates AS (
    SELECT * FROM {{ ref('seed_movie_release_date') }}
)
```

So now:

```text
your seed = just like any other table
```

Then you joined:

```sql
fct_ratings + seed_dates
```

---

# 4. What your query is doing (simple)

```sql
SELECT 
    f.*,
    CASE
        WHEN d.release_date IS NULL THEN 'unknown'
        ELSE 'known'
    END
```

### Meaning:

* If movie has release date → "known"
* If not → "unknown"

---

# 5. Why seeds are used (important)

Used for **small static data**

Examples:

* country codes
* product categories
* mapping tables
* test/demo data

---

# 6. Now difference from other things (VERY IMPORTANT)

## Seeds vs Sources vs Models

---

### 1. Seeds

```text
Input: CSV (local file)
Output: Table in warehouse
Use: Small static data
```

---

### 2. Sources

```text
Input: External data (S3, production DB, raw tables)
Output: Just reference (DBT does NOT create it)
Use: Raw data
```

Example:

```sql
SELECT * FROM {{ source('netflix', 'movies') }}
```

DBT is saying:

> “This table already exists, I’m just using it”

---

### 3. Models

```text
Input: SQL
Output: Table/View
Use: Transform data
```

Example:

```sql
SELECT * FROM {{ ref('fct_ratings') }}
```

---

# 7. Full Flow (your project)

```text
S3 → Raw tables (source)
        ↓
Staging models (cleaning)
        ↓
Dim + Fact models
        ↓
Seed (extra mapping data)
        ↓
Mart (final business table)
```

---

# 8. Key Difference (super simple)

| Thing  | Creates Table? | Data comes from |
| ------ | -------------- | --------------- |
| Seed   | Yes            | CSV file        |
| Source | No             | Existing DB/S3  |
| Model  | Yes            | SQL logic       |

---

# 9. One-line understanding

* **Seed** → “Upload CSV as table”
* **Source** → “Use existing data”
* **Model** → “Transform data”

---

# 10. Why this confused you

Because:

* Seed looks like model
* but actually it's **data loading tool**
* not transformation

---

# 11. Real-life analogy

Think like Excel:

* Seed → manually typed small table
* Source → imported data
* Model → formulas

---

### Short + clear difference

| Type          | What it is                  | Creates in DB? | Purpose                   |
| ------------- | --------------------------- | -------------- | ------------------------- |
| Seed          | CSV file → table            | Yes (table)    | Load small static data    |
| Table (model) | SQL result stored           | Yes (table)    | Final / heavy data        |
| View (model)  | Saved SQL query             | Yes (view)     | Light transformation      |
| Incremental   | Table (only new data added) | Yes (table)    | Big data optimization     |
| Ephemeral     | Temporary SQL (CTE)         | No             | Helper logic (not stored) |
| Source        | Existing table              | No             | Read raw data             |

---

### One-line each

* Seed → load CSV
* Table → store full data
* View → run query on demand
* Incremental → update only new rows
* Ephemeral → temporary logic
* Source → already exists data
