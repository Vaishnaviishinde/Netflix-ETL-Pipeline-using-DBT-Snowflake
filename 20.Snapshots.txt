20.Snapshots 

---

# 1. What is Snapshot (simple concept)

### One line:

> **Snapshot = track history of data changes over time**

---

## Normal table problem

Suppose your table:

```text
user_id = 18
tag = "Mark Waters"
```

Then later it changes to:

```text
user_id = 18
tag = "Mark Waters Returns"
```

### What happens normally?

```text
Old value → LOST ❌
```

---

## Snapshot does this instead:

```text
Row 1 → Mark Waters (old)
Row 2 → Mark Waters Returns (new)
```

So:

> **Nothing is deleted, everything is stored**

---

# 2. Why we use it (REAL purpose)

Used when:

* data keeps changing
* but you want history

Example:

* customer address change
* product price change
* user status change

# 4. What your snapshot config means

```sql
config(
    target_schema='snapshots',
    unique_key=['user_id','movie_id','tag'],
    strategy='timestamp',
    updated_at='tag_timestamp',
    invalidate_hard_deletes=True
)
```

---

## Simple meaning:

### 1. target_schema

```text
Where snapshot table will be stored
→ snapshots schema
```

---

### 2. unique_key

```text
How to identify a row uniquely
```

Here:

```text
user_id + movie_id + tag
```

---

### 3. strategy = timestamp

```text
Check changes using timestamp column
```

---

### 4. updated_at

```text
Which column tells latest update
→ tag_timestamp
```

---

### 5. invalidate_hard_deletes

```text
If row deleted → mark as inactive
```

---

# 5. What happens when you run

```bash
dbt snapshot
```

DBT does:

1. Compare current data vs previous snapshot
2. If change found →

   * keep old row
   * insert new row

---

# 6. Output table (important)

Snapshot creates extra columns:

```text
dbt_valid_from
dbt_valid_to
```

---

## Example

| user_id | tag                 | dbt_valid_from | dbt_valid_to |
| ------- | ------------------- | -------------- | ------------ |
| 18      | Mark Waters         | 2023           | 2025         |
| 18      | Mark Waters Returns | 2025           | NULL         |

---

### Meaning:

* First row → old value
* Second row → current value

# 9. What is surrogate key (simple)

```text
Artificial unique ID
```

You created:

```sql
dbt_utils.generate_surrogate_key(...)
```

Why?

Because:

```text
user_id alone not unique
```

So combine:

```text
user_id + movie_id + tag → unique row
```

# 1. How you checked snapshot (in short)

### Step 1: Check snapshot (before change)

```sql
select * from snapshots.snap_tags
where user_id = 18;
```

You saw:

```text
Mark Waters
```

---

### Step 2: Update source data

```sql
update src_tags
set tag = 'Mark Waters Returns',
    tag_timestamp = cast(current_timestamp() as timestamp_ntz)
where user_id = 18;
```

Now source table changed.

---

### Step 3: Verify source changed

```sql
select * from dev.src_tags 
where user_id = 18;
```

You saw:

```text
Mark Waters Returns
```

---

### Step 4: Run snapshot

```bash
dbt snapshot
```

---

### Step 5: Check snapshot again

```sql
select * from snapshots.snap_tags
where user_id = 18;
```

Now you saw:

```text
Row 1 → Mark Waters (old)
Row 2 → Mark Waters Returns (new)
```

---

### Final logic (one line)

**Update source → run snapshot → snapshot stores both old + new**

# 2. Interview Questions (Tricky / Hard)

### 1.

**Q:** If I update a row but don’t change `updated_at`, will snapshot detect change?

**A:** ❌ No
Because timestamp strategy depends only on `updated_at`.

---

### 2.

**Q:** What happens if `unique_key` is wrong?

**A:**

* DBT can’t track rows correctly
* duplicates / wrong history created

---

### 3.

**Q:** Difference between `timestamp` vs `check` strategy?

**A:**

* timestamp → checks `updated_at` column
* check → compares actual column values

---

### 4.

**Q:** Why snapshot instead of incremental model?

**A:**

* Incremental → updates latest data only
* Snapshot → keeps full history

---

### 5.

**Q:** What happens if a row is deleted in source?

**A:**
If `invalidate_hard_deletes = true` →

* DBT sets `dbt_valid_to` (marks inactive)

---

### 6.

**Q:** Why do we need surrogate key in snapshot?

**A:**
Because natural keys may not uniquely identify versioned rows →
surrogate key ensures unique tracking of each version.

---


Here are **5 real scenario-based, tricky snapshot questions (with answers)** — these are the ones interviewers use to catch people:

---

### 1. Data changed but snapshot didn’t capture it

**Q:**
You updated a column value in source, ran `dbt snapshot`, but no new row was created. Why?

**A:**
Most likely:

* `updated_at` column did not change
* or wrong column used in `updated_at`

**Key point:**
Timestamp strategy ONLY checks `updated_at`, not actual data.

---

### 2. Duplicate rows appearing in snapshot

**Q:**
Your snapshot table has duplicate records for same row. What went wrong?

**A:**

* Wrong `unique_key`
* Not truly unique combination

**Fix:**
Use proper key or surrogate key.

---

### 3. Snapshot growing too large

**Q:**
Snapshot table size is increasing rapidly. Why?

**A:**

* Frequent updates in source
* Every change creates new row

**Insight:**
Snapshot stores full history → size always grows.

---

### 4. Old records not getting closed (dbt_valid_to is NULL)

**Q:**
You see multiple active rows (dbt_valid_to = NULL). Why?

**A:**

* `updated_at` not updating properly
* DBT not detecting change

**Correct behavior:**
Only latest row should have `NULL`.

---

### 5. Row deleted in source but still exists in snapshot

**Q:**
You deleted a row in source, but snapshot still shows it. Why?

**A:**
Because snapshot keeps history.

**Extra:**

* If `invalidate_hard_deletes = true` → marked inactive
* If false → stays as-is

---

### One-line takeaway

**Snapshot problems = 90% due to wrong `updated_at` or `unique_key`**
