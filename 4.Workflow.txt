4.Workflow

project goal
main focus of the project is to learn and practice dbt transformations. data loading and infrastructure are kept simple so that most of the time is spent on dbt models.

dataset used
dataset used is movielens which simulates a netflix-like platform.

dataset idea
it contains movie related information such as
user ratings
movie details
tags added by users
movie genres
external movie links

example

ratings.csv

user_id | movie_id | rating | timestamp
12 | 300 | 4.5 | 2024-01-01

movies.csv

movie_id | title | genre
300 | avengers | action

purpose
this dataset helps simulate a streaming platform analytics scenario like netflix where we analyze ratings, movies, users etc.

overall pipeline used in project

data source → s3 → snowflake → dbt transformations

example flow

movielens dataset
↓
upload to aws s3
↓
load into snowflake tables
↓
transform using dbt models

three ways to load data into snowflake

approach 1
upload file directly from snowflake ui

local csv → snowflake upload → table created

limitations
file size limit around 250mb
not suitable for large production pipelines
not automated

approach 2 (used in project)
upload files to aws s3 then load into snowflake

csv → s3 bucket → snowflake copy command → tables

advantages
handles large files
more scalable
closer to real production pipelines
easier automation

approach 3
use dbt seed

local csv inside dbt project → dbt seed → table created in warehouse

limitations
meant only for small reference datasets
not for large data ingestion
not commonly used for raw production data

why approach 2 is used

1. scalable
   large files like gb or tb can be loaded from s3 easily.

2. common real world architecture
   many companies store raw data in data lakes like s3.

3. automation friendly
   snowflake can automatically ingest new files from s3.

4. closer to real data engineering pipelines

example real pipeline

application logs
↓
stored in aws s3
↓
loaded into snowflake
↓
dbt transforms data
↓
dashboards / analytics

why not approach 1

direct upload is mostly for testing or very small datasets. it is not practical for real pipelines where data arrives continuously.

why not approach 3

dbt seed is intended for small static datasets like lookup tables.

example

country codes
product category mapping
small config tables

not suitable for big datasets like movie ratings.

why not s3 directly with dbt

dbt cannot process raw files directly from s3. dbt works only on tables inside a data warehouse.

so first the data must exist inside a warehouse like

snowflake
bigquery
redshift

then dbt runs sql transformations on those tables.

example

wrong idea

s3 csv → dbt transform

correct pipeline

s3 csv
↓
snowflake table
↓
dbt transformations

why snowflake is used in this project

1. free 30 day trial
2. very easy to set up
3. widely used cloud data warehouse
4. integrates well with dbt
5. good skill to add on resume

but in real companies alternatives may be used like

Snowflake
Google BigQuery
Amazon Redshift

all of them can work with dbt.
