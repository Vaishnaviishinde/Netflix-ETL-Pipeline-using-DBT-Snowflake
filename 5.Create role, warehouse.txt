5.Create role, warehouse 

---

use role accountadmin;

what it does
switches to the highest privilege role in the system.

why needed
only admin roles can create users, roles, warehouses, and grant permissions.

example
if you try

create user dbt;

without admin role → permission error.

so first we switch to admin.

do companies do this
yes, but usually only platform engineers or senior data engineers run admin commands.

---

create role if not exists transform;

what it does
creates a custom role called transform.

role meaning
a role is a permission group.

example

role = job position

analyst role → read data
engineer role → create tables
admin role → full control

so here

transform role = role responsible for transformations (dbt).

why create role instead of using admin
security.

never run pipelines using accountadmin.

if dbt runs using admin and credentials leak → attacker gets full system access.

so create limited role.

companies always do this.

example real company roles

analytics_role
etl_role
bi_role
admin_role

---

grant role transform to role accountadmin;

what it does
allows accountadmin to use transform role.

basically admin can manage that role.

not extremely important for pipelines but useful for control.

---

create warehouse if not exists compute_wh;

what it does
creates a compute warehouse.

important concept

in Snowflake

warehouse = compute engine.

storage and compute are separate.

warehouse runs queries.

example

select * from movies;

runs on warehouse.

without warehouse → query cannot execute.

example analogy

database = data storage
warehouse = cpu processing queries.

do we do this every time
no.

companies usually create warehouses once.

example

etl_wh → for ingestion jobs
dbt_wh → for transformations
bi_wh → for dashboards

separate warehouses prevent workloads from affecting each other.

---

grant operate on warehouse compute_wh to role transform;

what it does
gives transform role permission to run queries using that warehouse.

without this

dbt user cannot execute queries.

example error

insufficient privileges to operate warehouse.

companies always explicitly grant warehouse permissions.

---

create user if not exists dbt

password='dbtPassword123'
login_name='dbt'

what it does
creates a user called dbt.

purpose
dbt tool will connect to snowflake using this user.

why create separate user

never connect tools using personal login.

bad practice

vaishnavi_login → dbt → pipelines run

problem

if you leave company → pipelines break.

correct

dbt_service_user → pipelines run.

companies use service accounts.

example real service users

dbt_user
airflow_user
fivetran_user

---

must_change_password=false

means first login does not force password change.

useful for automated services.

otherwise pipeline connection fails.

---

default_warehouse='compute_wh'

sets default compute warehouse when dbt connects.

so dbt does not need to specify warehouse every time.

example

dbt runs model

it automatically runs using compute_wh.

---

default_role=transform

whenever dbt connects → it automatically uses transform role.

so dbt gets correct permissions.

---

default_namespace='movielens.raw'

namespace meaning

database.schema

so

movielens = database
raw = schema

default namespace means

if dbt runs

create table movies;

it automatically creates

movielens.raw.movies

without writing full path.

---

comment='DBT user used for data transformation'

just documentation.

companies heavily comment service users for clarity.

---

alter user dbt set type = legacy_service;

what it does

marks this as a service account.

used for automated tools.

not always required but common in pipelines.

---

grant role transform to user dbt;

this connects the user to the role.

relationship

user → role → permissions.

example

dbt user
↓
transform role
↓
permissions to warehouse + database.

without this dbt user has no permissions.

---

create database if not exists movielens;

creates database for project.

database hierarchy in snowflake

account
→ database
→ schema
→ tables

example

movielens database
raw schema
movies table.

companies usually create databases per domain.

example

finance_db
marketing_db
product_db

---

create schema if not exists movielens.raw;

schema = logical folder inside database.

example

movielens database

raw schema
staging schema
analytics schema

raw = raw ingested data
staging = cleaned data
marts = final analytics.

dbt projects often follow this structure.

---

grant all on warehouse compute_wh to role transform;

gives transform role full control over warehouse.

so dbt can

run queries
start warehouse
stop warehouse.

---

grant all on database movielens to role transform;

gives transform role permission on database.

without this dbt cannot create tables.

---

grant all on all schemas in database movielens to role transform;

gives access to all existing schemas.

example

movielens.raw
movielens.staging
movielens.analytics

transform role can access all.

---

grant all on future schemas in database movielens to role transform;

very important line.

future schemas = schemas created later.

example

today

raw schema exists.

tomorrow

analytics schema created.

without this permission → dbt cannot access new schema.

companies always use future grants.

---

grant all on all tables in schema movielens.raw to role transform;

allows dbt to read/write existing tables.

---

grant all on future tables in schema movielens.raw to role transform;

another important line.

future tables = tables created later.

example

new table loaded tomorrow

ratings_2026

transform role automatically gets access.

without this → pipelines break.

companies always add future table grants.

---

big picture of what this code is doing

1 create role for transformations
2 create compute warehouse
3 create service user for dbt
4 create database + schema
5 give required permissions
6 allow future tables access.

this is basically **snowflake access setup for dbt pipelines**.

companies absolutely follow similar patterns, just with more strict roles and security policies.

---

small real world example

company setup

roles

ingestion_role
transform_role
analytics_role

users

airflow_user → ingestion_role
dbt_user → transform_role
bi_tool_user → analytics_role

each tool only gets permissions it needs.

this is called

principle of least privilege.
