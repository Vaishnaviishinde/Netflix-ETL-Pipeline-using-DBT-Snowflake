7. Why Create a Transform Role?

You asked a very good question:

if we give admin access then why not just use admin?

Because security isolation.

Example scenario.

Imagine this pipeline:

dbt → connects to Snowflake

If dbt logs in as ACCOUNTADMIN, then dbt can:

delete databases

delete users

drop tables

change security

So if something goes wrong or credentials leak:

Everything can be destroyed.

Instead companies do this:

dbt user
   ↓
transform role
   ↓
limited permissions

Example permissions:

create tables

update tables

run queries

But cannot delete users or databases.

This is called:

Principle of least privilege.

Every cloud system does this.

AWS example

You would NEVER run production systems with:

AdministratorAccess

Instead you create a role like:

EC2_App_Server_Role

with only required permissions.

Same logic here.

8. Why Give Admin Access to the Role Then?

Good catch — but look carefully.

He didn’t give full admin access.

This line:

GRANT ROLE TRANSFORM TO ROLE ACCOUNTADMIN;

means:

Admin can use the transform role, not the other way around.

Admin always has control anyway.

So this just allows admin to manage that role.

It does NOT give transform admin powers.

COMPUTE:

1. What is “Compute”?

Compute = CPU + RAM that executes work

Example work:

running SQL queries

filtering data

joining tables

calculating sums/averages

Basically the machine that does the calculations.

Example with Your Laptop

Your laptop has:

CPU

RAM

When you run Python:

df.groupby("movie").mean()

your laptop CPU processes the data.

So:

Laptop CPU = compute.

3. Cloud Version of This

In the cloud, your laptop doesn’t process the data.

Instead cloud machines do it.

Example:

AWS

Service	Compute
EC2	runs programs
EMR	runs Spark jobs
Lambda	runs functions

All of these = compute resources.

4. Now Snowflake

In Snowflake there are two separate things:

1️⃣ Storage (where data lives)
2️⃣ Compute (machines that process queries)

The compute part is called a Warehouse.

So:

Snowflake Storage → stores tables
Warehouse → runs queries

Example:

movies table stored in Snowflake
↓
SELECT * FROM movies
↓
Warehouse CPU processes query
5. Simple Real Example

Imagine a restaurant.

Storage = fridge
Compute = chef

Example:

fridge → stores ingredients
chef → cooks food

Mapping:

Snowflake Storage → fridge
Warehouse → chef

If no chef is working → no food gets cooked.

Same way:

If warehouse is OFF → queries cannot run.

6. Why Snowflake Does This

Traditional databases combine everything.

Example:

Postgres

data + compute on same server

But Snowflake separates them.

So you can scale compute independently.

Example:

Small warehouse → slow queries
Large warehouse → fast queries
