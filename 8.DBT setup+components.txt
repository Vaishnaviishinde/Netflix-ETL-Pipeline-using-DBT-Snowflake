8.DBT setup+components 

pip install virtualenv

command

```
pip install virtualenv
```

what this command does
This installs a Python tool called virtualenv. It lets you create an isolated Python environment for a specific project.

why we use it
When you install Python packages normally, they install globally on your system. If different projects need different versions of packages, they start conflicting.

Example problem without virtual environment

project A needs
dbt 1.5

project B needs
dbt 1.9

If both install globally, one will break.

virtual environment solves this by creating a separate package environment only for that project.

All installed libraries stay inside that folder.

important concept
Almost every real data engineering project uses virtual environments.

---

python -m venv venv

command

```
python -m venv venv
```

what this command does
Creates a virtual environment named venv inside the project folder.

after running it, a folder appears

```
venv/
```

inside it there are things like

```
venv
 ├── Scripts or bin
 ├── python
 ├── pip
 └── libraries
```

why we use it
This environment contains its own Python interpreter and package manager.

So when we install packages like dbt, they do not affect the system Python.

example

project structure after creating environment

```
netflix_dbt_project
 ├── venv
 └── (project files)
```

everything installed later will go into this venv folder.

---

source venv/bin/activate

command

```
source venv/bin/activate
```

windows equivalent

```
venv\Scripts\activate
```

what this command does
Activates the virtual environment.

why we use it
After activation, any Python packages installed will go inside the venv folder instead of the global system.

when activated the terminal usually changes like this

```
(venv) user@computer
```

That (venv) shows the environment is active.

example

before activation

```
pip install dbt
```

installs globally.

after activation

```
pip install dbt
```

installs inside

```
venv/lib/python/site-packages
```

important
Always activate environment before installing packages.

pip install dbt-snowflake==1.9

command

```
pip install dbt-snowflake==1.9
```

what this command does
Installs dbt core and the Snowflake adapter.

things installed

```
dbt-core
snowflake connector
dbt-snowflake adapter
```

why we use it
dbt itself is only a transformation tool. It needs a database adapter to communicate with a database.

Different databases require different adapters.

examples

Snowflake

```
dbt-snowflake
```

BigQuery

```
dbt-bigquery
```

Redshift

```
dbt-redshift
```

Postgres

```
dbt-postgres
```

In this project we are using Snowflake so we install dbt-snowflake.

example flow after installation

```
dbt
 ↓
snowflake adapter
 ↓
snowflake warehouse
```

dbt will send SQL queries to Snowflake through this adapter.

mkdir ~/.dbt

command

```
mkdir ~/.dbt
```

what this command does
Creates the root dbt configuration folder.

why we use it
dbt stores database connection configuration here.

inside this folder dbt stores

```
profiles.yml
```

this file tells dbt how to connect to the database.

example

```
~/.dbt
   └── profiles.yml
```

important idea
project code lives in the project folder
database credentials live in the dbt root folder

---

dbt init netflix

command

```
dbt init netflix
```

what this command does
Creates a new dbt project.

dbt generates a project structure automatically.

after running this command you will see

```
netflix
 ├── dbt_project.yml
 ├── models
 ├── macros
 ├── seeds
 ├── snapshots
 ├── tests
 └── analyses
```

why we use it
dbt projects follow a structured architecture. Instead of writing random SQL files, dbt organizes transformations into models and workflows.

example workflow

```
raw data → staging models → analytics models
```

dbt manages these transformations automatically.

---

profiles.yml

file location

```
~/.dbt/profiles.yml
```

what this file does
Stores database connection configuration.

example

```
netflix:
  target: dev
  outputs:
    dev:
      type: snowflake
      account: xyz123
      user: dbt
      password: ***
      role: transform
      database: movielens
      warehouse: compute_wh
      schema: raw
```

why we use it
Whenever dbt runs, it reads this file to know

which database to connect
which warehouse to use
which schema to run models in

so connection flow becomes

```
dbt project
 ↓
profiles.yml
 ↓
snowflake connection
```

---

dbt_project.yml

file location

```
project/dbt_project.yml
```

this is the main configuration file of the dbt project.

what it does
Controls project settings like

project name
model locations
materialization settings

example

```
name: netflix_project
version: 1.0

models:
  netflix_project:
    staging:
      materialized: view
    marts:
      materialized: table
```

meaning

staging models will create views
analytics models will create tables

why this is important
dbt uses this file to control how SQL models behave.

---

models folder

location

```
models/
```

this is the most important folder in dbt.

what models are
models are simply SQL files that transform data.

example model

```
models/staging/stg_movies.sql
```

example SQL

```
select
 movieId,
 title,
 genres
from raw_movies
```

dbt will run this SQL and create a table or view in Snowflake.

example flow

```
raw_movies table
 ↓
dbt model SQL
 ↓
stg_movies table
```

why models exist
Instead of writing transformations in Python scripts or manual SQL, dbt organizes them as modular SQL models.

real companies use multiple layers

```
raw
 ↓
staging
 ↓
intermediate
 ↓
mart
```

example

```
models
 ├── staging
 │     └── stg_movies.sql
 ├── intermediate
 │     └── movie_ratings_join.sql
 └── marts
       └── top_movies.sql
```

dbt automatically runs models in correct order based on dependencies.

---

macros folder

location

```
macros/
```

macros are reusable SQL functions written in Jinja.

what macros do
They allow reusable SQL logic.

example macro

```
macros/cents_to_dollars.sql
```

```
{% macro cents_to_dollars(column) %}
({{ column }} / 100)
{% endmacro %}
```

then inside model

```
select
 {{ cents_to_dollars("price") }} as price_usd
from sales
```

why macros exist

without macros you would repeat logic everywhere.

macros help create reusable SQL utilities.

very commonly used in

dynamic SQL generation
looping through columns
data cleaning functions

seeds folder

location

```
seeds/
```

seeds contain static CSV files.

dbt loads them directly into the warehouse.

example

```
seeds/country_codes.csv
```

example data

```
code,name
US,United States
IN,India
UK,United Kingdom
```

command

```
dbt seed
```

this loads the CSV as a table.

why seeds exist
Some data is small reference data that doesn't come from pipelines.

examples

country lists
currency codes
product categories

seeds allow version-controlled static tables.

---

snapshots folder

location

```
snapshots/
```

snapshots track historical changes in tables.

example problem

customer table

```
customer_id name city
1           John NY
```

later city changes

```
customer_id name city
1           John LA
```

if you overwrite table, history is lost.

snapshots preserve history.

example snapshot

```
customer_id name city valid_from valid_to
1           John NY   2024       2025
1           John LA   2025       NULL
```

why companies use snapshots

to track slowly changing dimensions.

example use cases

customer address history
employee salary changes
product price changes

tests folder

location

```
tests/
```

dbt can test data quality.

example test

```
not_null
unique
```

example

```
models/stg_movies.yml
```

```
columns:
  - name: movieId
    tests:
      - unique
      - not_null
```

this ensures

movieId is unique
movieId is never null

why this matters
dbt adds data validation inside the pipeline.

analyses folder

location

```
analyses/
```

not part of production models.

example

analyst testing a query

```
analyses/movie_analysis.sql
```

full dbt architecture now looks like this

```
S3
 ↓
Snowflake raw tables
 ↓
dbt staging models
 ↓
dbt intermediate models
 ↓
dbt marts
 ↓
BI dashboards
```

dbt acts as the transformation layer.

---

very important concept about dbt

traditional pipeline

```
python ETL scripts
```

dbt pipeline

```
SQL transformation DAG
```

dbt automatically builds dependency graph between models.

example

```
raw_movies
 ↓
stg_movies
 ↓
top_movies
```

dbt understands this dependency and runs models in correct order automatically.

---

once this part is understood, the next major commands become

```
dbt run
dbt test
dbt seed
dbt snapshot
```

these are the commands that actually execute the dbt pipeline.
